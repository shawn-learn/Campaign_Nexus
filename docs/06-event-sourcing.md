# 8. Event-Sourcing Design

Baseline decision: **hybrid event log**, not pure event sourcing — see ADR-004 for the
trade-off analysis. This document specifies the mechanics.

## 8.1 The write path (command pipeline)

Every mutation in the system flows through one pipeline. There is no second way to write.

```
API route
  └─► Command handler (module service)          ── one SQLite transaction ──┐
        1. authorize (campaign scope, role)                                  │
        2. validate command (Pydantic + domain invariants)                   │
        3. mutate current-state tables                                       │
        4. append DomainEvent row(s)  (seq = campaign counter++)             │
        5. run synchronous projection handlers for those events              │
        6. COMMIT ◄──────────────────────────────────────────────────────────┘
        7. publish committed events on the in-process bus (post-commit hooks)
```

- Steps 3–5 are atomic: state, event, and projections can never disagree (NFR-2.1).
- Step 7 runs **after commit** so subscribers never observe uncommitted state. Post-commit
  subscribers are *reactive conveniences* (story-suggestion evaluation, cache invalidation);
  anything correctness-critical belongs in step 5.
- Commands are plain service functions (`relocate_npc(...)`, `advance_time(...)`), not a
  command-bus framework — the pipeline is a decorator/context-manager (`with command_tx(...)`),
  keeping ceremony near zero for a solo developer.

## 8.2 Event anatomy

```json
{
  "id": "0197a2...",
  "campaign_id": "0196ff...",
  "seq": 4812,
  "event_type": "npc_relocated",
  "occurred_at_game": 302400,
  "recorded_at_real": "2026-07-07T02:15:09Z",
  "session_id": "0197a1...",
  "actor": "time_engine",
  "payload": {
    "npc_id": "0196aa...", "from_location_id": "0196bb...",
    "to_location_id": "0196cc...", "reason": "schedule:weekly_market"
  },
  "narrative_text": "Serah Voss traveled from Duskmere to Harrowgate (weekly market)."
}
```

Design rules:
1. **Past tense, facts only.** Events record what happened, never what should happen next.
2. **Dual clocks.** `occurred_at_game` (campaign minutes — FR-5.7) and `recorded_at_real`.
   Backdating is legal (GM records lore after the fact); `seq` preserves recording order.
3. **`narrative_text` is mandatory** — generated by the command handler from the payload.
   It is the timeline's display text and the future AI corpus (ADR-012). Structured payload
   for machines, sentence for humans, written once at the moment of maximum context.
4. **Entity references** are mirrored into `event_entity` rows for indexed filtering.
5. **Append-only.** Mistakes are corrected by compensating events
   (`event_retracted {retracts: seq}` plus, where needed, a corrective fact). Projections
   honor retractions; the log never lies about what was once believed.

## 8.3 Event catalog (initial)

| Family | Events (payload highlights) |
|---|---|
| Time | `time_advanced (from, to, reason)`, `scheduled_event_fired (sched_id)` |
| Party | `party_moved (from, to, method)`, `party_traveled (route, duration)`, `short_rest_completed`, `long_rest_completed`, `downtime_spent (activity, duration)`, `gold_changed (delta, reason)`, `item_acquired/lost`, `reputation_changed (faction, delta)` |
| NPC | `npc_relocated`, `npc_status_changed (alive→dead …)`, `npc_met_party`, `npc_interaction (kind, summary)`, `npc_learned (topic_entity)` |
| Quest | `quest_revealed/available/accepted/completed/failed/expired/abandoned`, `quest_objective_done (index)` |
| Combat | `combat_started (encounter_id, combatants)`, `combat_ended (rounds, duration_game, outcome)`, `combatant_defeated (entity, by)`, `combat_summary (casualties, notable_moments[])` |
| Session | `session_started (number)`, `session_ended`, `note_captured (text)` |
| World/Story | `flag_changed (key, old, new)`, `story_node_activated/resolved/abandoned`, `world_event (title, description)` — the generic carrier for festivals, invasions, harvests |
| Wiki | `entity_created/updated/deleted/restored`, `link_added/removed` — audit-level; **not** timeline material by default |
| Discovery | `secret_revealed (entity, secret_ref)`, `treasure_discovered (items, value)`, `location_discovered` |

The catalog is a versioned enum in code; each event type declares a Pydantic payload model
(validation) and a `significance` default (timeline projection input).

## 8.4 Projections

All synchronous, all rebuildable. Handler registry: `event_type → [handlers]`.

| Projection | Source events | Notes |
|---|---|---|
| `timeline_entry` | every event with `significance ≥ threshold` + manual entries | GM can hide entries (curation) without touching facts |
| `npc.current_location_id`, `npc_location_history` | `npc_relocated` | history row closed (`to_game`) when the next relocation opens one |
| `npc.has_met_party`, `last_party_interaction_game` | `npc_met_party`, `npc_interaction` | |
| `party.current_location_id` | `party_moved/traveled` | |
| `party` gold/inventory/reputation | `gold_changed`, `item_*`, `reputation_changed` | |
| Session auto-links (FR-9.3) | any event with `session_id` | materialized as `GROUP BY` view at read time — no table needed |
| Quest status columns | `quest_*` | status column is authoritative-looking but event-derived |
| Search index | `entity_created/updated/deleted` | contentless FTS5 upsert |

**Rebuild:** `nexus rebuild-projections <campaign>` truncates all projection tables and
replays `domain_event` in `seq` order through the same handlers. Used for: recovery, handler
bug fixes, schema evolution of projections, and as the **consistency oracle in tests**
(property: incremental result ≡ rebuild result).

## 8.5 Combat: the pure event-sourced island (ADR-005)

Combat needs unlimited undo/redo (FR-12.4) — the one requirement that genuinely wants pure
event sourcing.

- `combat_action` rows are the log; **combat state is never stored per-field** (only a
  snapshot cache at the cursor for fast reload).
- `state = fold(actions[0..cursor])` — a pure reducer in Python, mirrored by an identical
  TypeScript reducer for optimistic UI (NFR-1.3). Reducer parity is contract-tested with
  shared JSON fixtures.
- **Undo** = `cursor--`, **redo** = `cursor++`; a new action while `cursor < max(seq)`
  deletes the redo tail then appends. All cursor ops are single-row updates — instant.
- Rules-plugin involvement: the reducer is system-agnostic (HP, conditions, turn order as
  data); the plugin supplies initiative sorting, condition catalog, and any auto-effects
  (e.g., concentration break prompts) as *suggestions*, keeping the reducer deterministic.
- **On `end combat`:** the run folds to final state → command emits `combat_ended`,
  `combatant_defeated`×n, `combat_summary` domain events → clock advances
  `rounds × round_length` (plugin constant) → timeline gets one entry, not three hundred.

## 8.6 Event schema evolution

- Payload models are versioned implicitly by `event_type` + additive-only fields; breaking
  changes mint a new type (`npc_relocated_v2`) with an upcast function old→new registered in
  the replay path. Upcasts run at read/replay time; stored rows are never rewritten.
- `narrative_text` is stored, not re-derived — old events keep reading correctly even if
  generation templates change.

## 8.7 What is deliberately *not* event-sourced

Wiki article content (`entity.article_json`) is versioned by `entity_updated` audit events
carrying content hashes, not full diffs — full text history is a post-1.0 feature
(snapshot table), and storing every keystroke in the domain log would bury the narrative
signal. The domain log records *world facts*, not editing activity minutiae.
